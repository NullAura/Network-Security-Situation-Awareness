# 实践代码说明

本文档用于帮助第一次接触本项目的人快速理解代码结构、数据链路和各脚本之间的关系。主 README 偏向系统功能与部署说明，本文更偏向“代码怎么读、每个文件负责什么、数据如何从日志变成前端页面”。

## 1. 项目目标

本项目实现的是一个多源蜜罐威胁情报大数据实践系统。系统从多个蜜罐节点采集安全日志，在日志产生端做预处理，然后通过 Flume/Kafka/Flink/HDFS 完成数据导入、流式处理、历史存储和前端可视化。

核心目标有三点：

1. 把 Cowrie、OpenCanary、自定义低交互蜜罐等异构日志统一成一套事件模型。
2. 用 Kafka、Flink、HDFS 串起实时采集、消息缓冲、计算处理和历史存储链路。
3. 用前端页面展示态势总览、攻击分析、源 IP 画像和数据导出结果。

## 2. 推荐阅读顺序

如果只是想快速理解代码，建议按下面顺序阅读：

1. `README.md`：了解系统功能、页面、整体架构和运行方式。
2. `实践代码说明.md`：了解每个目录和关键文件的作用。
3. `infra/honeypot_edge_preprocessor.py`：理解边缘预处理如何把原始日志转成统一 JSONL。
4. `infra/flume-*.conf`：理解预处理后的日志如何进入 Kafka。
5. `infra/flink/honeypot-flink-stream/src/main/java/edu/bigdata/honeypot/HoneypotFlinkStreamJob.java`：理解 Flink 如何消费 Kafka 并写入流式文件。
6. `infra/honeypot_flink_hdfs_sync.sh` 与 `infra/preprocess_honeypot_logs.py`：理解流式文件如何同步到 HDFS、如何生成 dashboard 快照。
7. `infra/honeypot_static_server.py` 与 `infra/honeypot_hdfs_export_api.py`：理解前端 API、CSV 导出和 HDFS 只读导出。
8. `frontend/src/services/threatApi.js`：理解前端如何加载、标准化和缓存后端数据。
9. `frontend/src/App.jsx` 与 `frontend/src/styles.css`：理解页面布局、图表、地图和源 IP 画像交互。

## 3. 总体数据链路

```mermaid
flowchart LR
  A["蜜罐原始日志"] --> B["边缘预处理"]
  B --> C["Flume Taildir"]
  C --> D["Kafka Topic"]
  D --> E["Flink 消费"]
  E --> F["本地 stream JSONL"]
  F --> G["HDFS ODS/DWD"]
  G --> H["dashboard 快照/API"]
  H --> I["React 前端"]
```

链路可以拆成六层：

| 层级 | 主要代码 | 作用 |
|---|---|---|
| 蜜罐采集层 | `infra/honeypot3/honeypot3.py`、Cowrie/OpenCanary 服务配置 | 产生 SSH、HTTP、VNC、Telnet、数据库、Redis 等攻击日志。 |
| 边缘预处理层 | `infra/honeypot_edge_preprocessor.py` | 监听本机日志新增行，统一字段、时间、协议、攻击方式。 |
| 消息缓冲层 | `infra/flume-*.conf`、`infra/setup_kafka_kraft.sh` | Flume 监听预处理 JSONL 并写入 Kafka Topic。 |
| 流式计算层 | `infra/flink/.../HoneypotFlinkStreamJob.java` | Flink 消费 Kafka，保留 topic/partition/offset 并写入 stream 文件。 |
| 存储与聚合层 | `infra/honeypot_flink_hdfs_sync.sh`、`infra/preprocess_honeypot_logs.py` | 上传 ODS/DWD 到 HDFS，生成 dashboard、summary 和 cleaned_events。 |
| 展示与导出层 | `infra/honeypot_static_server.py`、`infra/honeypot_hdfs_export_api.py`、`frontend/src/*` | 发布 API、导出 CSV、展示态势总览和源 IP 画像。 |

### 3.1 按服务器划分的代码与配置

线上部署时，不是把整个仓库原样复制到每台机器，而是按服务器角色分发对应脚本、配置和 systemd 服务。下面按当前系统节点说明每台服务器需要关注的代码。

#### 3.1.1 `203.0.113.10`：Cowrie SSH 蜜罐节点

原始日志：

```text
/home/cowrie/cowrie/var/log/cowrie/cowrie.json
/home/cowrie/cowrie/var/log/cowrie/cowrie.json.YYYY-MM-DD
```

部署到该节点的项目文件：

| 仓库文件 | 线上用途 |
|---|---|
| `infra/honeypot_edge_preprocessor.py` | 本地预处理 Cowrie JSON 日志，统一时间、IP、协议、攻击方式、账号口令和 payload 字段。 |
| `infra/honeypot-edge-preprocessor.service` | 以 systemd 方式持续运行边缘预处理脚本。 |
| `infra/flume-cowrie.conf` | Flume Taildir 配置，监听预处理后的 JSONL 文件。 |
| `infra/honeypot-flume.service` | 启动 Flume Agent，把 Cowrie 预处理事件写入 Kafka。 |

该节点主要环境变量或配置项：

| 配置项 | 含义 |
|---|---|
| `LOG_PATH` | Cowrie 当前日志路径。 |
| `LOG_GLOB` | Cowrie 当前日志和轮转日志匹配规则。 |
| `OUTPUT_PATH` | 预处理 JSONL 输出路径，通常是 `/var/log/honeypot-preprocessed/events.jsonl`。 |
| `STATE_PATH` | 预处理 offset 状态文件，避免重启后重复读取。 |
| `HONEYPOT_TYPE` | 固定为 `cowrie`。 |
| `SERVER_ID` / `SERVER_IP` | 统一写入事件来源标识。 |
| `kafka.bootstrap.servers` | Kafka 节点地址。 |
| `kafka.topic` | 写入 `honeypot_cowrie_clean`。 |

运行结果：

```text
Cowrie 原始日志
  -> honeypot_edge_preprocessor.py
  -> /var/log/honeypot-preprocessed/events.jsonl
  -> Flume Taildir
  -> Kafka topic: honeypot_cowrie_clean
```

常用检查命令：

```bash
systemctl status honeypot-edge-preprocessor
systemctl status honeypot-flume
tail -n 5 /var/log/honeypot-preprocessed/events.jsonl
```

## 5. 核心数据模型

预处理后的事件统一为 JSONL，每行是一条事件。常见字段如下：

| 字段 | 含义 |
|---|---|
| `schema_version` | 预处理事件模型版本。 |
| `server_id` / `server_ip` | 事件来源节点。 |
| `raw_log_path` | 原始日志路径。 |
| `raw_time` | 原始日志里的时间字段。 |
| `event_ts_utc` | UTC 时间。 |
| `event_ts_beijing` | 北京时间。 |
| `event_ts_ms` | 毫秒级时间戳，用于排序、窗口统计和 HDFS 分区。 |
| `src_ip` | 攻击源地址。 |
| `dst_ip` / `dst_port` | 目标地址与端口。 |
| `protocol` | 协议或服务类型。 |
| `event_type` | 原始事件类型。 |
| `attack_method` | 攻击方式，例如弱口令爆破、指纹探测、SQL 注入探测等。 |
| `username` / `password` | 登录尝试相关字段。 |
| `command` | 命令输入。 |
| `payload` / `payload_hex` | 请求内容、异常输入或十六进制载荷。 |
| `country` / `region` / `city` / `isp` / `asn` | 地理位置和网络归属字段。 |

## 6. 关键代码说明

### 6.1 边缘预处理

文件：`infra/honeypot_edge_preprocessor.py`

职责：

- 持续读取蜜罐本机日志新增行。
- 用 `state-path` 保存 inode 和 offset，避免重启后重复读取或漏读。
- 解析 Cowrie JSON、OpenCanary JSON 行、自定义蜜罐 JSONL。
- 统一时间、协议、源地址、目标端口、事件类型和攻击方式。
- 输出到 `/var/log/honeypot-preprocessed/events.jsonl` 这类预处理 JSONL 文件。

关键函数：

| 函数 | 作用 |
|---|---|
| `parse_time()` | 兼容多种时间格式。 |
| `event_time()` | 从原始字段中挑选事件时间，失败时使用采集时间。 |
| `protocol_from()` | 根据显式字段、事件类型或端口推断协议。 |
| `method_for()` | 根据协议、事件类型、payload、账号口令和命令推断攻击方式。 |
| `normalize_event()` | 将一条原始日志转换为统一事件。 |

阅读重点：

- 看 `METHOD_LABELS` 和 `SQLI_MARKERS` 可以理解攻击方式识别规则。
- 看 `PORT_PROTOCOLS` 可以理解端口到协议的映射。
- 看主循环可以理解“监听文件新增行 + 保存 offset”的实现。

### 6.2 Flume 到 Kafka

文件：

- `infra/flume-cowrie.conf`
- `infra/flume-opencanary.conf`
- `infra/flume-honeypot3.conf`
- `infra/honeypot-flume.service`

职责：

- 使用 Flume Taildir Source 监听预处理输出文件。
- 使用 file channel 做本地缓冲。
- 使用 Kafka Sink 写入对应 Topic。

三个主要 Topic：

| Topic | 含义 |
|---|---|
| `honeypot_cowrie_clean` | Cowrie SSH 预处理事件。 |
| `honeypot_opencanary_clean` | OpenCanary 多协议预处理事件。 |
| `honeypot3_clean` | 自定义低交互蜜罐预处理事件。 |

阅读重点：

- `filegroups` 决定监听哪些文件。
- `positionFile` 决定 Flume 自己保存的读取位置。
- `kafka.bootstrap.servers` 和 `kafka.topic` 决定写入哪个 Kafka。

### 6.3 Flink 流式消费

文件：`infra/flink/honeypot-flink-stream/src/main/java/edu/bigdata/honeypot/HoneypotFlinkStreamJob.java`

职责：

- 订阅三个清洗后的 Kafka Topic。
- 保留 Kafka `topic`、`partition`、`offset`、`timestamp`。
- 将每条 Kafka 消息包装为 stream JSONL。
- 按日期和来源写入本地目录，供后续同步到 HDFS。

主要类：

| 类 | 作用 |
|---|---|
| `JobOptions` | 解析 Flink 作业参数。 |
| `KafkaPollingSource` | 用 KafkaConsumer 拉取消息并交给 Flink。 |
| `StreamFileSink` | 按日期写入本地 JSONL 文件。 |
| `KafkaEnvelope` | 封装 Kafka 元信息和原始消息。 |

阅读重点：

- `AUTO_OFFSET_RESET` 当前默认从最新消息开始消费，重算历史时需要调整消费策略。
- 输出 stream 文件不是最终 DWD，只是 Kafka 消息落地后的中间层。

### 6.4 HDFS 同步与批量清洗

文件：

- `infra/honeypot_flink_hdfs_sync.sh`
- `infra/preprocess_honeypot_logs.py`

职责：

- 把 Flink stream 文件上传到 HDFS ODS。
- 调用批量清洗脚本生成 DWD 明细。
- 生成 `preprocess_summary.json`、`dashboard.json`、`dashboard-live.json` 等前端需要的快照。
- 把 DWD 明细写入 HDFS 日期分区。

HDFS 目录约定：

```text
/honeypot/ods/stream/YYYY-MM-DD/*.jsonl
/honeypot/dwd/cleaned_events/date=YYYY-MM-DD/cleaned_events.jsonl
/honeypot/dwd/cleaned_events/date=YYYY-MM-DD/preprocess_summary.json
```

`preprocess_honeypot_logs.py` 的重点：

| 逻辑 | 说明 |
|---|---|
| 文件遍历 | 支持输入文件或目录，跳过临时文件和备份文件。 |
| 事件加载 | 兼容外层 Flink 包装和内层预处理事件。 |
| 时间处理 | 生成 UTC、北京时间和毫秒时间戳。 |
| 地理位置 | 使用缓存和外部查询补全 country、region、city、isp、asn。 |
| 攻击方式 | 重新归一化协议和 attack_method。 |
| dashboard | 生成前端态势总览、攻击分析、源 IP 画像所需数据。 |

### 6.5 前端 API 和导出

文件：

- `infra/honeypot_static_server.py`
- `infra/honeypot_hdfs_export_api.py`
- `frontend/server/honeypot_static_server.py`

职责：

- 发布静态前端文件。
- 发布 dashboard 快照。
- 提供运行状态探针。
- 代理 HDFS 导出接口。
- 按时间范围、协议、攻击方式和字段列表导出 CSV。

关键点：

- `honeypot_static_server.py` 更靠近前端 API，负责加载快照、过滤事件、导出 CSV 和健康探针。
- `honeypot_hdfs_export_api.py` 更靠近存储节点，只读读取 HDFS DWD 分区并生成 CSV。
- 导出接口使用 token 和来源限制，避免任意用户读取 HDFS 路径。

### 6.6 实时 dashboard

文件：

- `infra/honeypot_realtime_dashboard.py`
- `infra/honeypot_live_dashboard.py`

职责：

- 周期性读取最新 stream 或 dashboard 数据。
- 生成前端实时窗口数据。
- 发布 `dashboard-live.json` 或刷新 `dashboard.json`。

阅读重点：

- 实时窗口不等于历史总量。最近 1 分钟没有新事件时，实时窗口可以为 0。
- 前端展示的历史事件数来自历史快照或 DWD 聚合，不应该和实时窗口混淆。

### 6.7 前端数据适配

职责：

- 定义协议、端口、攻击方式、蜜罐节点、导出字段等元数据。
- 控制缓存时间和展示限制。

阅读重点：

- `fetchDashboard()` 和相关加载函数决定前端数据来源。
- `normalize*` 系列函数决定不同后端字段如何被前端识别。
- `DISPLAY_SOURCE_IP_LIMIT` 等常量只影响展示窗口，不应影响后端全量导出。

## 9. 配置与安全约束

阅读或修改代码时要注意以下约束：

1. 不要把服务器口令、私钥、API token 写入仓库。
2. Kafka、Flink、HDFS 管理端口不应对公网开放。
3. HDFS 导出 API 只允许读取固定 DWD 根目录，不允许用户传入任意 HDFS 路径。
4. 作业提交接口不应暴露给普通访问者。
5. 前端导出应走受控 API，不直接暴露 HDFS 命令。
6. 采集端只做本地预处理和日志推送，不承担 Hadoop/Flink 作业提交。

部分脚本里包含部署默认值，迁移环境时应通过命令行参数、systemd 环境变量或配置文件覆盖，避免把某个实验环境的地址写死到新环境。

## 11. 排查问题的顺序

如果前端数据不对，建议按链路从上到下检查：

1. 蜜罐原始日志是否有新增行。
2. 预处理输出 JSONL 是否新增。
3. Flume position 是否推进。
4. Kafka Topic 是否有消息。
5. Flink stream 目录是否生成当天 JSONL。
6. HDFS ODS/DWD 分区是否有当天数据。
7. `dashboard.json` / `dashboard-live.json` 是否更新。
8. 前端浏览器是否读取了最新 API，而不是缓存旧文件。

如果某一层没有数据，优先检查该层的 systemd 状态、日志、文件路径和权限，再检查上一层是否真的有输出。
